# Kaleid Scan 要件定義

## 機能要件

### 0. 画面・ナビゲーション構成

> **画面設計参照**: `docs/screen-design/Home.png`

- アプリはスキャン画面の**1画面のみ**で構成する
- 画面遷移は「スキャン画面 → 商品詳細ボトムシート（オーバーレイ）」のみ

### 1. カメラスキャン

| 項目 | 内容 |
|------|------|
| カメラ | バックカメラを使用 |
| 撮影方式 | シャッターボタン押下で静止画を取得 |
| ローディング | AI解析中はローディング演出を表示 |

### 2. AI画像認識

| 項目 | 内容 |
|------|------|
| 使用API | ローカル: Gemini 2.0 Flash Vision API / 本番: AWS Bedrock（Claude Sonnet） |
| 識別方式 | 静止画をバックエンド経由でAI APIに送信 |
| 識別対象 | コンビニ商品5種（おにぎり×2・飲み物×2・お菓子×1） |
| 返却情報 | 商品名 + バウンディングボックス（座標）|
| 複数商品 | 画面内の全商品を同時識別。最大5商品 |
| 未検出時 | エラーモーダル「商品が検出できませんでした」を表示 |

### 3. オーラ表示

| 項目 | 内容 |
|------|------|
| デザインコンセプト | 商品の周囲に広がる柔らかい円形ハローグロー |
| 表示位置 | 商品の背後・周囲（商品本体には被せない） |
| 強弱 | 1位：大きく明るい / 5位：小さく薄い（5段階） |
| 複数対応 | 検出された全商品にオーラを付与 |

> **画面設計参照**: `docs/screen-design/Home.png`

**オーラレベル定義**

| ランク | オーラレベル | 見た目イメージ |
|--------|------------|--------------|
| 1位 | Lv.5 | 金・暖色・大きな円形ハロー |
| 2位 | Lv.4 | 青・寒色・やや大きめ |
| 3位 | Lv.3 | 緑・中程度 |
| 4位 | Lv.2 | 紫・小さめ |
| 5位 | Lv.1 | グレー・薄く控えめ |

### 4. 商品詳細（ボトムシート）

| 項目 | 内容 |
|------|------|
| 表示方式 | スキャン画面のオーラをタップすると、下から競り上がるボトムシートで表示（独立画面ではない） |
| 商品画像 | 撮影画像をバウンディングボックスでトリミングした写真を中央に表示 |
| 表示情報 | ランキングバッジ（例：🏅 1位）・商品名・商品説明 |
| 商品説明 | チームが手動作成（5商品分） |
| 閉じる | 上部のドラッグハンドルを下スワイプで閉じる |

> **画面設計参照**: `docs/screen-design/Item-Detail.png`
> モックアップ下部に「X で共有」ボタンが表示されているが、これは **Phase 3（X共有機能）** として先行でデザインに含まれているもの。MVPでは実装しない。

### 5. SNS共有（X）※ MVP後に実装

| 項目 | 内容 |
|------|------|
| 対応SNS | X（Twitter）のみ |
| 共有方法 | Web Share API（navigator.share）でXへ画像+テキストを渡す |
| 画像生成 | Canvas APIでオーラ合成画像をフロントエンド側で生成（サーバー不要） |
| 投稿文言 | `KaleidScanで〇〇を発見しました！ #KaleidScan` |
| 備考 | S3へのアップロード不要。モバイルブラウザ前提 |

### 6. PWA対応

| 項目 | 内容 |
|------|------|
| ホーム画面追加 | Web App Manifest を設定 |
| カメラアクセス | HTTPS環境での getUserMedia 対応 |
| Service Worker | オフライン対応は最低限（ハッカソンスコープ外） |

## 非機能要件

| 項目 | 目標値 |
|------|--------|
| **レスポンス時間** | スキャン〜オーラ表示まで 2秒以内 |
| **対応環境** | iOS Safari / Android Chrome（モバイル前提） |
| **API使用量** | ローカル: Gemini 2.0 Flash 無料枠内（1,500 req/日）。本番: Bedrock 従量課金 |

## 技術スタック

### ローカル開発環境

| レイヤー | 技術 |
|----------|------|
| フロントエンド | React + TypeScript（Vite）+ PWA |
| バックエンド | Go + Gin |
| データベース | PostgreSQL（Docker） |
| AI | Gemini 2.0 Flash Vision API（無料枠） |
| 環境管理 | Docker Compose |

### AWS 本番環境

| レイヤー | 技術 |
|----------|------|
| フロントエンド | React（Amplify Hosting） |
| バックエンド | Go（Lambda + API Gateway） |
| データベース | PostgreSQL（RDS） |
| AI | AWS Bedrock（Claude Sonnet） |
| CI/CD | GitHub Actions + IaC |

> **設計方針**: 環境変数の差し替えだけでローカル → AWS 本番に切り替えられるよう設計する。

## アーキテクチャ概要

```
[Mobile Browser (PWA)]
        |
        | HTTPS
        v
[API Gateway]
        |
        v
[Lambda（Go + Gin）]
        |          \
        |           \
        v            v
[RDS (PostgreSQL)]   [ローカル: Gemini / 本番: Bedrock]
```

**スキャンの処理フロー**

```
1. ユーザーがシャッターを押す
2. フロントエンドが画像を取得（静止画）
3. 画像をバックエンドへ送信（multipart/form-data）
4. バックエンドが AI（ローカル: Gemini / 本番: Bedrock）へ画像 + プロンプトを送信
5. AI が商品名 + バウンディングボックスを返す
6. バックエンドが商品名でDB検索し売上ランキングを付与
7. バックエンドが結果（商品情報 + 座標 + ランク）をJSONで返却
8. フロントエンドがCanvas APIで各商品の座標にオーラを描画
```

**SNS共有のフロー**

```
1. ユーザーが「共有」ボタンを押す
2. フロントエンドがCanvas APIでオーラ合成画像を生成（ローカル処理）
3. navigator.share({ files: [imageFile], text: "..." }) でXへ渡す
4. Xアプリが起動し、画像＋テキストが添付された投稿画面が開く
```

## 技術的判断の根拠

| 判断 | 採用 | 理由 |
|------|------|------|
| **AI（ローカル）** | Gemini 2.0 Flash | Gemini 1.5 Pro は無料枠50 req/日。ローカル開発中のテスト+デモで枯渇リスクが高い。Flash は1,500 req/日で現実的 |
| **AI（本番）** | AWS Bedrock（Claude Sonnet） | 事前の概念検証でうまくいきそうだったから |
| **X共有の画像添付** | Web Share API | `navigator.share({ files })` でS3アップロード不要。モバイルブラウザ（iOS Safari / Android Chrome）で画像ファイルをXへ直接渡せる |
| **DB** | PostgreSQL（RDS） | 週次売上集計・ランキング計算にSQLが適切。DynamoDB は集計クエリが複雑になるため不採用 |
| **フロントホスティング** | Amplify Hosting | React PWA + GitHub連携のCI/CDが最も手間なく構築できる |
| **商品ID型** | UUID | 将来の商品追加・外部連携時の安全性を考慮。この規模ではSERIAL整数でも十分だが、設計変更コストを考えUUIDを維持 |
